MODULE PROJECT RUBRIC

Project Topic

You may choose any domain for your NLP project. Topic selection is intentionally open to encourage creativity and real-world relevance.

Important restriction:
You may not reuse a project you are already working on for another course, prior research, or your job. This project must be new work developed specifically for this course.

 

Project Overview & Requirements

Your project should align with the themes and techniques covered in this module (computer vision), though you are encouraged to extend beyond them if appropriate. The goal is to develop, evaluate, and deploy a model that predicts or analyzes real-world data, while demonstrating sound ML engineering, evaluation rigor, and usability.

 

Required Modeling Approaches

You must implement and evaluate three modeling approaches:

Naive baseline
Example: mean predictor, majority class classifier, random baseline
Classical (non–deep learning) ML model
Example: linear/logistic regression, random forest, gradient boosting, SVM
Neural network–based deep learning model
You may select any one of these approaches as your final deployed model.
However, all three must be implemented and included in your repository, with clear documentation explaining where each can be found.

 

Required Experimentation

You must conduct at least one focused experiment that probes your modeling choices. Examples include (but are not limited to):

Sensitivity analysis (e.g., training set size vs. performance)
Robustness testing (e.g., noise injection, distribution shift)
Comparison of preprocessing pipelines
Ablation studies
Any other experiment in which the results are meaningful to developing and testing your system
Your experiment should be well-motivated, clearly described, and properly interpreted.

 

Interactive Application Requirement

Your project must include a working, interactive proof-of-concept application that is:

Publicly accessible via the internet
Live for at least one week after submission
Running model inference only (no training in the app)
You may use any frontend or deployment framework (e.g., Flask, Streamlit, Dash; AWS, GCP, Azure).

UX matters.
A minimal or poorly designed interface will be penalized. This is not just a model demo, it is a product-level proof of concept. Note: a basic Streamlit app is not going to cut it.

If the app does not run when graded, you will receive a 0 for this portion of the project.
Think of this as an investor demo: if it doesn’t work, it doesn’t get funded.

 

Project Approach

Links to an external site.
Your goal with the project is to create a new, unique contribution to solving the problem which goes above and beyond anything which has been done previously. There are two options on how to approach this for the project:

You may choose to work on a dataset or problem for which there are no identifiable previous modeling approaches (and therefore, your project will be entirely novel), e.g. if you create a new dataset for the project or no one has previously attempted to model the dataset.

For example, suppose you create a new text dataset about chemistry by gathering text from many articles about areas of chemistry, and train a new NLP topic model to extract the key topics of each article
Or perhaps you gather hundreds of images of different types of birds and then train an image classification system to recognize bird species
Alternatively, you may choose to work on an existing dataset or problem which others have worked on (e.g. you find a dataset on Kaggle). If you do work on an existing problem, you must explain and show what is new and novel about your approach relative to previous approaches. You should also make sure to reference the previous approaches you reviewed. Your approach should either result in state-of-the-art (SOTA) performance or close to it, OR provide greater insight into the problem (e.g. better explainability) as compared to previous approaches.

For example, suppose you find an open source dataset of medical chest xray images and several references of disease classification models using the data. You take a different approach to modeling it by using a computer vision model to extract certain features from the images and then feeding those into a decision tree model.

Project repo structure

Links to an external site.
Your project repo should be organized similar to the below structure. You may deviate from the below structure if you wish, but make sure you have at least the main components (README, requirements.txt, main.py, helper scripts, model(s), data). You may also add additional files/directories if needed, e.g. dockerfile, database or html/css.

├── README.md               <- description of project and how to set up and run it
├── requirements.txt        <- requirements file to document dependencies
├── Makefile [OPTIONAL]     <- setup and run project from command line
├── setup.py                <- script to set up project (get data, build features, train model)
├── main.py [or main.ipynb] <- main script/notebook to run project / user interface
├── scripts                 <- directory for pipeline scripts or utility scripts
    ├── make_dataset.py     <- script to get data [OPTIONAL]
    ├── build_features.py   <- script to run pipeline to generate features [OPTIONAL]
    ├── model.py            <- script to train model and predict [OPTIONAL]
├── models                  <- directory for trained models
├── data                    <- directory for project data
    ├── raw                 <- directory for raw data or script to download
    ├── processed           <- directory to store processed data
    ├── outputs             <- directory to store any output data
├── notebooks               <- directory to store any exploration notebooks used
├── .gitignore              <- git ignore file

Code Quality Rules

Jupyter notebooks are allowed only in the notebooks/ directory (Note: we will NOT look at Jupyter notebooks for grading - these are for your research and exploration only)
All code must be modularized into classes and functions
No loose executable code outside functions or if __name__ == "__main__"
Use descriptive variable names, docstrings, and comments
If you use external code (including AI usage), you must attribute it at the top of the file with a link to the original source
